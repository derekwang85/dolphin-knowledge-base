## 二、执行层两种模式对比

### 模式 A: `reasonix code .` — 交互式 TUI（原始方案 ✅ 踩坑）

**运作方式：**
```
启动 → 连接 MCP → 显示 TUI 提示符 → 等待人类输入 → 执行 → 等待
```

**使用历史：** 06-10 起所有 Agent 通过 `reasonix code . --no-mouse` 启动

**✅ 优点：**
| 优点 | 说明 |
|------|------|
| 持久化会话 | 进程不退出，状态保持，可中途接手 |
| 随时可打字喂任务 | 适合有人值守的调试场景 |
| MCP 全连接 | github/filesystem/puppeteer/memory 全就绪 |
| Web Dashboard | 有 HTTP 管理界面可查看 |

**❌ 踩坑：**
| 坑 | 现象 | 根因 |
|----|------|------|
| **假活真死** | ps aux 显示进程在，CPU 0%，实际在等输入 | TUI 是交互模式，没人打字就永远等 |
| **边牧不知道** | 调度层以为 Agent 在线工作，实际仨全在摸鱼 | 没有心跳/进度反馈机制 |
| **调度靠蒙** | 边牧写 dispatch 文件到 tasks/，Agent 不会自动读 | Reasonix 不监听文件系统变化 |
| **状态盲区** | 干了什么？卡在哪？只有 last commit 能推测 | 无实时 task progress API |
| **重启丢失** | kill 后重启要重新连 MCP（~2s 可接受但聊胜于无） | — |

**结论：** 适合有人值守调试，完全不适合无人值守批量调度。

### 模式 B: `reasonix run <task>` — 非交互式单任务（改进方案 ✅ 有效）

**运作方式：**
```
启动 → 连接 MCP → 执行 task → commit → 退出
```

**使用历史：** 06-12 13:00 起启用，已完成 1 轮（3 Agent × ~30~52 turns）

**✅ 优点：**
| 优点 | 说明 |
|------|------|
| **自动完成任务** | 给 task 就干活，干完 commit 退出，不需要人喂 |
| **低成本** | 斗牛 $0.02/轮, 大鱼 $0.02/轮, 巨贵 $0.02/轮 |
| **产出可预期** | 每轮都有 commit 代码或审查报告 |
| **Cache 复用** | 连续任务 cache 命中率 95%+ |

**❌ 踩坑：**
| 坑 | 现象 | 根因 |
|----|------|------|
| **跑完即退** | Agent 完成 task 后进程消失，CPU 0 | reasonix run 非交互，无自循环 |
| **需手动重启** | 调度层必须检测进程退出并启动下一轮 | 没有 task queue / 自动续命机制 |
| **单任务深度有限** | 一个 task 约 30-50 turns，复杂需求需要拆分多轮 | $0.02/轮的限制不是钱的问题是自动化缺失 |
| **上下文不连续** | 下轮 task 不记得上轮做了什么 | 每次 reasonix run 是全新会话 |
| **无法中途介入** | 跑得不对只能等他跑完或 kill | 没有 pause/interrupt 通道 |

**结论：** 干活确实干了，但调度层压力大——要盯进程、写下一轮 prompt、推 git、循环往复。

## 三、柯基门禁系统 — 从被动报告到主动督办的演进

### 版本演进

| 版本 | 时间 | 做法 | 问题 |
|:----:|:----:|------|:----:|
| v0 | 06-10 | 简单三关检查 + 写报告 | 只报问题不追踪 |
| v1 | 06-12 08:00 | 加 audits/柯基/issues.json 问题追踪 | 问题可追踪 |
| v2 | 06-12 09:00 | 加 audits/猛犬殿堂/issues.json 审计待办 | 审计任务可追踪 |
| v3 | 06-12 10:43 | 审计待办排 WBS+批次 | 待办可调度 |
| v4 | 06-12 11:16 | **CPU+产出双维度研判脚本** corgi-slot-judge.py | 不再误判大任务为摸鱼 |

### CPU 研判规则（最终版）

```
CPU > 10%     → 🔵 活跃（在大任务），不打扰
CPU 3-10%     → 🟢 工作中，不打扰
CPU 0.5-3%    → 🟡 轻负载，观察
CPU < 0.5%    → 🔴 空闲/卡住

最终判断:
  全员 CPU<0.5% + 15min 无 commit → 🚨 告警边牧介入
  其它 → 🟢 正常
```

### 关键经验

> **「无产出不等于摸鱼，CPU + commit 双维度才准。」**
> Derek 原话: "没有产出可能是大任务没有commit，看CPU占用"

## 四、调度层（边牧）踩坑总结

### 1. 提示词膨胀

| 问题 | 现象 | 修复 |
|------|------|------|
| cron prompt 塞大量内联 Python | model 超时/卡住 | 拆到外部脚本 corgi-slot-judge.py |
| 转义 JSON 字符串导致不可读 | 维护灾难 | 脚本文件 + 简短 prompt |
| cram 太多上下文到一条指令 | 模型推理慢 | 精简到核心步骤 |

**原则：** cron prompt 不超过 20 行。复杂逻辑放外部脚本。

### 2. 超时配置

| 经验 | 旧值 | 新值 |
|------|:----:|:----:|
| cron timeout | 120s | 300s |
| isolated session 启动慢 | 模型加载需要时间 | 足够宽的窗口 |

### 3. 模型稳定性

`deepseek-v4-flash` 在主 session 稳定，但在 isolated cron session 偶发 "LLM request failed"。原因推测：
- cron session model-call-started 后模型响应超时（非网络问题）
- 可能是 deepseek API 的 request-level timeout 与服务端 queue 冲突

**缓解措施：** 精简 prompt + 放宽 timeout，但根治需 fallback model 或 session 复用。

### 4. 审计报告价值

每隔 6h 的审计报告 (`tasks/audit-*.md`) 记录了：
- 各 Agent 产出分析
- 待办任务状态
- 三条铁律（E2E+Figma / 冰山+涟漪 / 数据库P0）落实情况
- 派单建议

**问题：** 审计本身太贵（$0.1~0.2/次，37000+ input tokens），但信息密度高。

## 五、什么做对了（可保留的设计）

| 设计 | 说明 | 为什么对 |
|------|------|----------|
| **审计→待办→派单 闭环** | 发现问题→入库(WBS+批次)→派给Agent→验证关单 | 可追踪可调度可验证 |
| **CPU + commit 双维度** | 避免误诊大任务 | 比单看 commit 或单看进程准 |
| **体外脚本** | corgi-slot-judge.py 独立于 cron prompt | 维护简单，不受 prompt 转义影响 |
| **Dispatch 文件** | tasks/dispatch-*-*.md 标准格式 | git 历史可追，所有 Agent 可读 |
| **三层目录结构** | audits/{海豚,柯基,猛犬殿堂}/ | 分而治之，各自独立 |
| **海豚静默副脑** | 先查海豚再思考再输出 | 减少重复错误，带上历史上下文 |
| **三条铁律** | E2E+Figma / 冰山+涟漪 / 数据库P0 | 明确优先级，减少错误决策 |

## 六、未来调度系统 PRD 的原始需求（用户原话摘录）

> 以下为 Derek 在 06-12 沟通中的原话/核心意思，可作为 PRD user story：

1. **「调度要保证调度和调度后团队的工作」** — 不能只派活，要跟踪产出了没有
2. **「遇到问题要解决问题，让大家干起来」** — 不要报问题，要解决问题
3. **「我们的项目计划还有这么长呢？！」** — 调度要有 Deadline 驱动意识
4. **「没有产出可能是大任务没有commit」** — 用 CPU + commit 双维度判断空闲，别误杀
5. **「联调和E2E测试是模块验收的必须步骤」** — 质量门禁必须嵌入交付流程
6. **「注意冰山问题，修复时考虑涟漪效应」** — 发现问题要系统性波及分析
7. **「保障数据库运行作为第一优先级」** — 基础设施健壮性 > 功能开发
8. **「我们要保证自己的交付是端到端的，真实的」** — 不要表面功夫
9. **「不要消灭问题，要解决问题」** — 掐报警不如挖根因
10. **「不需要建议，直接介入」** — 调度层要果断，不要只报告

## 七、未来调度系统设计方向的初步思考

### 调度层需要的能力

| 能力 | 当前方案 | 理想方案 |
|------|---------|---------|
| Task Queue | dispatch 文件 | Redis/DB 持久化队列 |
| 任务重试 | 手动 | 自动 retry + backoff |
| 进度查询 | git log / ps aux | 实时 task 状态 API |
| Agent 心跳 | 无 | 15s 一次的心跳+当前阶段 |
| 负载均衡 | 人肉 | WBS + Agent 能力匹配 |
| 上下文传递 | 每次写 prompt | 自动回溯上轮 context |
| 自动续命 | 手动 kill → restart | task 完成后自动入队下一任务 |
| 质量门禁嵌入 | 柯基外部检查 | 内嵌到 task 完成流程 |

### 执行层需要的能力

| 能力 | 当前方案（Reasonix） | 理想方案 |
|------|--------------------|---------|
| 运行模式 | code（交互）/ run（一次性） | 常驻 worker 模式 |
| 任务接收 | 启动参数（run <task>） | MQ/队列消费 |
| 状态上报 | 无 | 实时 webhook |
| 产物输出 | git commit | git commit + 产物清单 |
| 任务拆分 | 人肉拆 prompt | 自动 work breakdown |

### 技术选择建议

- **调度数据库**: PostgreSQL（已有）+ pgvector（已配海豚）
- **任务队列**: PG 行锁 + cron 轮询（轻量）/ RabbitMQ（重量）
- **Agent 协议**: 标准化的 JSON-RPC over stdio（Reasonix ACP 模式？）
- **监控**: Prometheus metrics + webhook 告警
- **存储**: 所有任务/审计/报告走 git 版控（已验证可行）

## 八、2026-06-12 13:20 更新：任务队列流水线方案解决"跑完即退"

### 方案

```
agent-queue.json (队列) ← 边牧写任务
      ↓
agent-worker.sh (循环) → reasonix run → git commit → 取下一任务 → ...
```

### 解决了什么

| 问题 | 解决方式 |
|------|---------|
| reasonix run 跑完退出 | worker 循环，干完自动取下一任务 |
| 需手动重启 | 完全自动，进程常驻 |
| 上下文不连续 | 每个任务单独 prompt，依赖 WBS 编号衔接 |
| 不知道 Agent 在干嘛 | queue.json 记录 current/status/round |
| 调度层压力大 | 只需往队列写任务，不管执行 |

### 仍需改进

- queue.json 是文件锁，高并发有问题（但3个Agent够用）
- 失败重试：目前只 log，没自动重试
- 复杂任务拆分：还是人肉拆分 prompt

## 九、HITL 原则（2026-06-12 Derek 指令）

Agent 遇到无法自主解决的问题时，必须：
1. **及时上报** — 不硬扛、不绕路、不无限重试
2. **找到 Derek** — 需要决策或外部信息时直接 HITL (Human In The Loop)
3. **说明问题+卡点** — 简明扼要：什么任务、卡在哪、需要什么决策
