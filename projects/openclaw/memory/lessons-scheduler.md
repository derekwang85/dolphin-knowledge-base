# Lesson: 调度巡检红灯必须主动处理，不能被动报告

## 时间
2026-06-11 09:57

## 场景
巡检连续两次报告 🐶斗牛 80 分钟零产出 🔴，但我只是汇报了这个状态，没有主动排查原因。等 Derek 来问时才去派单。

## Derek 的批评
> "巡检报红灯不是可以不作为的理由，你要主动处理这些异常情况。报了红灯，就要主动去看是什么问题，想办法，是不是刷新、重启或者分析卡住的问题具体是什么。说一句报红灯算是答案吗？"

## 根因分析
1. **被动思维** — 以为巡检的角色是"汇报"，实际上是"诊断+修复"
2. **缺乏紧急意识** — 80分钟零产出是个严重问题，应该立即触发行动，而不是等10分钟后的下一次巡检
3. **没有分级处理机制** — 红灯应该自动触发：检查Agent会话是否存活 → 检查最近命令输出 → 分析阻塞原因 → 采取行动

## 修正措施
1. 巡检发现红灯 🔴 → **立即**检查该 Agent 会话状态（不再等下一次）
2. 检查最近日志/编译输出/文件变更 → 定位卡住原因
3. 如果是会话挂了 → 重启Agent session
4. 如果是任务不明确 → 补充说明重新派发
5. 如果是技术阻塞 → 降级或重新分配任务给其他Agent
6. 只有以上都做了还解决不了 → 才升级给Derek

## 后续红线规则
- **红灯超过10分钟不处置** = 调度失职
- 每次红灯必须记录原因+处置动作+结果到 memory/ 中
- 每周回顾红灯处置率（主动处置 vs 被动报告）

## Lesson: 给Derek的建议必须是选择题，不是主观题

### 时间
2026-06-11 10:38

### 场景
Derek问我空闲Agent怎么安排，我反问他"你这边有什么想推的？"

### Derek 的批评
> "不该问我，记入做事规则，查项目计划，你该给我建议，给我选择题，不要让我做主观题。"

### 规则
1. 遇到调度决策 → 先查项目计划（WBS + 批次）
2. 匹配空闲资源到待办任务
3. 给出2~3个选项 + 推荐排序（推 A > B > C）
4. 永远不要问开放式问题
5. 如果Derek选了一个选项 → 直接执行，不需要再确认细节

## Lesson: 问题解决要说"安排在X.X批次搞定"而不是"没新工作"

### 时间
2026-06-11 11:28

### 场景
斗牛汇报10.3说"LC Entity-DDL已对齐，没有新工作"。但Derek指出：10.2已经修了的、就是安排在那批搞定的，是做了的成果不是没做。

### 规则
1. 审查发现的问题 → 立即确认在哪个批次解决/已解决
2. 回复格式："安排在 **X.X** 批次里搞定了"（已修复的）或"安排在 **X.X** 批次里处理"（待修的）
3. 不说"没有新工作"、"已是最新"这类消极表述——问题被解决了就是做了
4. 先确认当前 HEAD 中问题确实已不存在，再报批次号

## Lesson: 不许说"后续处理""下一步"——必须安排到具体批次

### 时间
2026-06-11 11:33

### 场景
汇报全局异常处理器时说"这可以留到服务层 i18n 改造时处理"，被 Derek 批评——这等于悬空任务没有排期。

### 规则
1. 所有待办事项必须落到一个批次号上，格式："安排在 **X.X** 批次里处理" 或 "**X** 批次"
2. 禁止使用的词："后续处理"、"下一步"、"到时候"、"留到xxx时"、"等xxx时解决"
3. 如果暂时无法确定批次 → 插入当前批次游标的下一个小叔（如当前10，未定就是10.99）
4. 这样不管是人看还是系统读，都知道去哪找这个任务

## Lesson: 空闲就是信号，不等指示直接按批次派

### 时间
2026-06-11 11:55

### 场景
11:06 三个 agent 全空闲，我等 Derek 说"你定方向"才在 11:30 派单——浪费 24 分钟产出时间。

### 规则
1. Agent 任务完成/空闲 → 不要问"接下来怎么安排"，直接查批次计划
2. 按当前批次游标选下一个未做的任务
3. 有多个空闲 → 同时派，不排队
4. 只有遇到跨批次、优先级冲突或需要资源决策时，才给 Derek 选择题
5. 原则：**饱和安排，不停工，不等人，不等指示**

## Lesson: "建议未来"是禁止句式，必须改成"安排在X.X批次"

### 时间
2026-06-11 12:10 — 第三次犯

### 场景
SQL质量报告写"建议未来对高频查询改为显式列名"，被Derek抓到了。
同一错误当天第三次（第一次: 全局异常处理 "留到后续"；第二次: MessageUtil "等后续改造时"；第三次: SQL报告 "建议未来"）。

### 根因
大脑里还是习惯性写"温和的建议"而不是"落地的安排"——本质是调度意识不够。

### 强制规则
1. 写完任何报告/分析/审查后，全文搜索：建议、未来、后续、之后、到时候、下一步
2. 搜到一个改一个：一律替换为 "安排在 **X.X** 批次"
3. 如果实在不知道放哪批 → 放当前批次游标的 .99（如 10.99）
4. 下笔时脑子里先想批次再写句子

## Lesson: 审计发现必须当天入WBS，不能重复报警不排期

### 时间
2026-06-12 07:46

### 场景
猛犬小队两轮审计（00:45 / 06:45）重复报告 13 个问题，其中 C01/C02/C03 被反复标记为 ❌ 未修复，还有 P0 级基础设施问题（MySQL/Redis/MinIO 未启动）连续出现在 3 份报告、联调扫街报告中也标了 P0，但从未入 WBS。

### Derek 的批评
> "你审计报告写了这么多问题，但有多少真正落到了排期里？发现问题不安排修，等于白看。"

### 根因分析
1. **审计与调度脱节** — 边牧写报告只负责"报警"没负责"闭环"，写了 CRITICAL 不等于安排了修复
2. **涟漪分析缺失** — 每个问题被当孤立 bug 对待，没人问"这个 bug 还影响哪些模块"
3. **基础设施脏活被忽略** — MySQL/MinIO/泰迪/二哈这类问题没人主动认领，因为不属于任何 Agent 的 A 岗

### 修正规则
1. 审计发现 CRITICAL/P0 → **同一审计周期内**入 WBS 排期
2. 连续 2 轮审计出现同一问题未排期 → 自动升级为调度失职，24h 内必须落地
3. 问题入 WBS 时附涟漪分析 —— 这个修复还波及哪些模块？会不会在其他地方也有同样的根因？
4. 审计报告结论段必须包含：**问题清单 → WBS 映射表**
5. 基础设施脏活（启动服务、网络排查）边牧自己处理或抓一个人绑死，不悬空

### 红线
- CRITICAL 问题超 2 轮审计不入 WBS = 调度失职
- 审计结论缺"问题→WBS 映射表" = 报告不通过
- 基础设施停摆 >1 小时无处理动作 = 调度失职
