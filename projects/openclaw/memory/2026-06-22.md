## 轨道 A — Provider 抽象层（架构进化）✅ 全部完成

### 问题
embedder.py / entity.py / reranker.py 都硬编码 Ollama，切换模型要改 3 个文件，不支持 vLLM/OpenAI/FreeCode，无 GPU 加速，无自动降级。

### 设计演化（Derek 指导核心决策）
1. **第一次设计**（被 Derek 否决）：按能力分裂的 3 个 Provider 接口（EmbeddingProvider / ExtractionProvider / RerankProvider）→ 太僵化，Provider 一次实现 3 个接口
2. **第二次设计**（被 Derek 指出不够灵活）：Model 作为配置单元 + Pipeline 策略（fallback/quality/speed）→ 缺少认知能力维度
3. **最终设计**：Model Registry（四维属性：intelligence/cost_tier/speed_tier/quality）+ Pipeline 约束求解器（min_intelligence / max_cost / max_latency_ms 约束 + objective 优化）+ ModelProfiler（自动探测模型位置）

### 实现代码
- **engine/model.py**（240行）— ModelEndpoint 统一接口 + ModelProfile 四维属性 + satisfies 约束检查
- **engine/registry.py**（200行）— ModelRegistry（注册/发现/自动 profile/磁盘缓存）
- **engine/pipeline.py**（200行）— PipelineResolver 约束求解（严格→relax→rank）
- **engine/profiler.py**（425行）— ModelProfiler（boot probe 3 samples 2s + deep probe 15 samples 10s）+ PassiveObserver（零成本侧效应监控）+ 震荡检测
- **engine/engine.py**（170行）— Engine facade（统一 embed/extract/rerank/generate API）
- **drivers/ollama.py**（230行）— OllamaModel（embed/extract/rerank/generate 四合一）
- **drivers/vllm.py**（110行）— VLLMModel（embed only，GPU）
- **drivers/openai.py**（180行）— OpenAIModel（兼容 OpenAI/FreeCode/MiniMax）
- **config.yaml**（195行）— 新格式（models + pipeline 取代旧三段）

### config.yaml 新格式
```yaml
models:
  bge-m3: {driver: ollama, capabilities: [embed], ...}
  qwen-7b: {driver: ollama, capabilities: [extract, rerank], intelligence: high, ...}
  bge-m3-gpu: {driver: vllm, capabilities: [embed], speed_tier: fast, ...}
  gpt-4o-mini: {driver: openai, capabilities: [extract, rerank], intelligence: very_high, cost_tier: paid_low, ...}
pipeline:
  embed: {requires: {max_cost: free}, objective: optimize_speed, candidates: [bge-m3-gpu, bge-m3]}
  extract: {requires: {min_intelligence: low, max_cost: free}, objective: optimize_quality, candidates: [qwen-7b, qwen-1.5b]}
```

### 核心设计决策（Derek 确认）
- Model 是配置第一公民，Provider 只是运输驱动
- 统一 invoke(capability, **kwargs) 接口取代 3 个分裂接口
- PipelineResolver 不是线性 fallback，是多维约束求解器
- Profiler 只用 golden data（LongMemEval），零外部 API 成本

### Profiler 设计要点
- Boot probe: 3 samples, ~2s, imprecision @0.5 上限
- Deep probe: 15 samples, ~10s, precision @0.85
- PassiveObserver: 不额外调 API，数据来自实际查询副产品（延时/实体数/错误率）
- 用量驱动 probe（50→100→200→400 次查询）
- 震荡检测：3 次 deep probe 内 intelligence 来回跳 → 冻结 + 报警
- 周期缓存 7 天
- FreeCode 场景：零配置，系统自动测速+定位

## 轨道 C — 实体→图谱改进 ✅ 全部完成

Derek 在 spec 评审中指出"由实体到图谱的改进漏了"——这是一个重要的方向性纠正。

### C1 — relation_inferrer.py（440行）
从代码文件中提取结构化关系，7 种确定性模式：
- `extends` — 类继承（`class X extends Y`）
- `implements` — 接口实现（`class X implements Y`）
- `depends_on` — 依赖注入（`@Autowired`, Maven 依赖）
- `is_a` — 类型标记（`@Service`, `@Component`）
- `serves_path` — API 路由（`@PostMapping("/api/x")`）
- `imports` — 模块导入（Java/Python/Vue import）
- `configures` — 配置引用（`${property.ref}`）

关系类型词表从 5 种扩展到 13 种。

### C2 — merge.py（90行）
跨模式实体/关系合并策略：
- 高精确率模式（regex=1.0, hybrid-7b=0.9）优先
- 低精确率模式（1.5b=0.5）保留但降权
- 冲突时高权重覆盖低权重

### C3 — entity_resolver.py（200行）
实体名解析与去重：
- 同义词典（KYCApplication = kycapplication = KYC 申请）
- 驼峰拆分（KYCApplicationService → KYC + Application + Service）
- 后缀剥离（Service/Impl/DTO/VO/DAO 等）
- Fuzzy match（trigram/levenshtein）

### C4 — graph_pruner.py（220行）
噪声清理：
- 47 个 stop words（data/get/set/code 等常见噪音）
- 稀疏实体（<2 条 chunk 链接）删除
- 孤儿实体（0 链接 + 0 关系 + 30 天以上）归档
- 1.5b-only 实体降权至 0.3

### C5 — enricher.py（170行）
实体描述丰富化：
- 离线 LLM 批量生成一句话描述
- 从关联 chunk 提取上下文
- 存入 entities.metadata.description
- CLI: `derekinside graph enrich`

### C6 — subgraph.py + CLI + HTTP（200行）
子图查询：
- BFS 遍历知识图，返回中心实体 + N 度邻居
- ASCII 树可视化
- CLI: `derekinside graph subgraph "KYC" --depth 2 --ascii`
- HTTP: `GET /api/v1/graph/subgraph?entity=KYC&depth=2`

### 集成到 graph_build
graph_build 的 chain 改为：
```
提取 → 关系推理 → 实体解析 → 建entity → 链接chunk → 存关系
```

## 架构级决策

### 1. SystemConfigService 模式（可配置化架构）

每个模块的可配参数注册到 `SystemConfigService`（Redis 缓存 + @CacheEvict），分组=模块前缀：
- group='credit' → 分档阈值/释放比例/组合因子
- group='kyc' → 有效期/AI开关/驳回次数
- 改完即时生效，无需重启

**核心**：模板驱动，不是 NL→SQL。OpenClaw 只做意图分类+参数提取，数据查询是预设模板。

### 2. 系统配置模块诞生（Admin 重构）

Admin 不再做业务操作，管系统配置+运行状态。8 个子模块（S1~S8）：
- S1 权限矩阵 | S2 用户管理 | S3 运行监控 | S4 审计日志
- S5 编号规则 | S6 功能配置 | S7 告警配置 | S8 状态管理 + S9 后台任务管理

606 行 spec，72 子任务，69 测试用例，14 个 FMEA。

### 3. DataPermission 数据权限模式

Ops 只能看自己被授权的 buyer/产品线/区域。CDP 存储 ops-buyer 映射，所有 OE 查询端点自动过滤。

### 4. OpenClaw 全系统 B 轨定位

OpenClaw 不限于 OE 驾驶舱的 NL 查询→整个系统的移动端对话代理，可调 28 模块。
