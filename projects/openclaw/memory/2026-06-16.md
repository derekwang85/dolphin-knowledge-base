## Bad 90 实战案例 (本场3条)

### Bad 90 #1 — 编译修复未做涟漪分析，逐文件修越修越多 (ISS-023)

**现象**: 看到114个编译错误，直接逐文件sed修复→暴露下一批→再修→再暴露，循环2小时。

**根因链**:
1. **核心错误**: 没先做冰山分析就动手。195个文件6月13日后被修改未编译验证，12个文件报错是冰山一角
2. **衍生问题1**: 每修一个文件暴露3-5个新错误 → KycApplicationService→CdpController→RegionAssembler→OrderAssembler→...
3. **衍生问题2**: 每次修复都要重新`mvn compile`等60秒 → 浪费大量时间
4. **系统性根因**: 没有`mvn compile`门禁 → Agent提交的代码全是未经编译验证的 → 195个文件坏了没人知道

**核心教训**: 任何问题第一反应必须是冰山三层分析。不看透影响面不动手。

### Bad 90 #2 — Runtime循环依赖用 @Lazy 打补丁 (ISS-024)

**现象**: Spring循环依赖(BusinessEventDispatcher↔BusinessEventProcessor↔OrderApplicationService)，尝试3次加@Lazy都失败，最后才用接口提取根治。

**根因链**:
1. **核心错误**: 把设计问题当配置问题修。循环依赖不是缺@Lazy注解，是EventProcessor直调OrderService.createFromQuote()的架构耦合
2. **衍生问题1**: 第1次加@Lazy→没加对→Derek帮忙
3. **衍生问题2**: 第2次换@ComponentScan→还是错→Derek再帮忙
4. **衍生问题3**: 第3次用`allow-circular-references=true`→不够→Derek又执行命令
5. **系统性根因**: BidOrderService接口提取是正确方案，应该第一轮就做

**核心教训**: Runtime问题要分析依赖图再动手。用接口隔离比用注解打补丁靠谱。HITL应该是决策不是操作。

### Bad 90 #3 — HITL违规 (ISS-025)

**现象**: 让Derek执行了3次终端命令（pkill、sed、mysql）。HITL应该是帮我做决策，不是替我操作。

**根因链**:
1. **核心错误**: exec超时→改用background→nohup日志看不到→需要Derek帮忙检查
2. **衍生问题**: 每次让Derek执行命令都打断他的思路，3次=浪费15分钟
3. **系统性根因**: 应该用`disown`+进程文件的方式分离后台任务，或者写一键启动脚本

**核心教训**: HITL=Human In The Loop. 人只在决策环节介入，操作环节不该碰触.

## 归纳总结

### Bad 90 在本阶段的3种典型表现

| 模式 | 症状 | 根治措施 |
|:-----|:------|:---------|
| **无涟漪直接修** | 修了A暴露B, 修了B暴露C | 任何问题先做冰山三层分析, L3根因修完才算完 |
| **注解打补丁代替重构** | @Lazy/@Autowired堆砌代替接口提取 | Runtime问题先画依赖图, 接口隔离>注解 hack |
| **HITL进操作层** | 让人执行终端命令 | 后台任务用disown/进程文件分离, 人只做选择(A/B/C) |

### 根因汇聚: 三者共同的系统性根因

```
1. 没有编译门禁 (mvn compile) — Agent离终端太远了，提交的代码未经验证
2. 没有构建前依赖检查 — 改了一个文件不知道牵连了谁
3. HITL契约不清晰 — 什么时候该找Derek决策 vs 什么时候该自己搞定
```
