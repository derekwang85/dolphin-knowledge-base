# ADR-005: 国际化（i18n）架构设计

> **状态**: 已接受 | **日期**: 2026-06-09 | **作者**: aITMS01

---

## 背景

OMS 面向大宗商品国际贸易场景，Buyer 端面向海外客户（英语、东南亚语系），运营端面向国内团队（中文）。核心需求：

- Buyer 端界面需支持多语言（至少中英双语）
- 报价/合同/LC 等文档需支持多语言版本
- 运营端内部使用中文，但需在 Buyer 可见字段上标注英文
- OpenClaw 对话需支持语言自适应（根据用户语言回复）

## 决策

### 决策 1：语言优先级与覆盖范围

| 语言 | 优先级 | Buyer 端 | 运营端 | 文档 | OpenClaw |
|------|--------|---------|-------|------|----------|
| 简体中文 (zh-CN) | P0 | ✅ | ✅ | ✅ 国内版 | ✅ |
| 英文 (en) | P0 | ✅ | 仅字段标注 | ✅ 国际版 | ✅ |
| 印尼语 (id) | P1 | ✅ | ❌ | 按需 | ⏸ 暂缓 |
| 越南语 (vi) | P1 | ✅ | ❌ | 按需 | ⏸ 暂缓 |
| 泰语 (th) | P2 | ⏸ | ❌ | ⏸ | ⏸ |

**第一阶段 (Phase 1-2)**: zh-CN + en 双语言
**第二阶段 (Phase 3+)**: 按需扩展东南亚语系

### 决策 2：技术实现方案

**选择 Java ResourceBundle + 数据库双轨方案**：

```
┌─────────────────────────────────────────────┐
│               翻译源体系                       │
│                                              │
│  UI 文本 → ResourceBundle (.properties)      │
│    ├── 后端: messages_zh_CN.properties       │
│    ├── 后端: messages_en.properties          │
│    └── 前端: i18n.json (Vue-i18n/React-i18n)│
│                                              │
│  业务数据 → 数据库多语言字段                   │
│    ├── oms_product (name_zh, name_en)        │
│    ├── oms_quote (terms_zh, terms_en)        │
│    └── oms_template (content_zh, content_en) │
│                                              │
│  模板文档 → 双语言模板                        │
│    ├── LC模板: 中英文版本分离                  │
│    └── 报价单: Content 字段可存 JSON          │
└─────────────────────────────────────────────┘
```

#### 分离原因

| 类别 | 方案 | 理由 |
|------|------|------|
| UI 静态文本 | ResourceBundle | 性能好，热加载，无 DB 查询 |
| 业务多语言字段 | 数据库字段 `_zh`/`_en` | 按行存储，支持按语言查询 |
| 模板文档 | 双模板 | 结构差异大，不适合字段级别 |

### 决策 3：业务数据多语言规范

**每个多语言实体遵循 `{field}_{locale}` 命名模式**：

```sql
-- 示例: oms_product 表
name_zh   VARCHAR(200) NOT NULL COMMENT '中文品名',
name_en   VARCHAR(200) NOT NULL COMMENT 'English product name',
desc_zh   TEXT COMMENT '中文描述',
desc_en   TEXT COMMENT 'English description',
```

**规则**：
1. 中文（zh）为默认必填，英文（en）为必填
2. 新增语系时加字段（`name_id`, `name_vi`），不修改已有字段
3. 查询时根据 `Accept-Language` 头返回对应语言字段
4. 若某个语言缺失值，回退到 zh

### 决策 4：语言协商机制

```
请求 → Accept-Language 头
  ├── zh-CN → 中文数据 + 中文UI
  ├── en → 英文数据 + 英文UI
  ├── id → 印尼语数据 + 英文UI（印尼语UI暂缓时回退英文UI）
  └── 未识别/缺省 → zh-CN（运营端）/ en（Buyer端）
```

**OpenClaw 语言自适应**：检测用户消息语言，用相同语言回复；首次交互用用户资料中预设语言。

### 决策 5：翻译管理流程

```
需求方(Product) 提供中文文案
  │
  ├── UI文案 → 写入 .properties / i18n.json
  │   ├── Phase 1: 人工翻译 → 直接嵌入代码
  │   └── Phase 2+: 接入翻译管理平台(TMS API) + LLM 辅助翻译
  │
  ├── 业务字段文案 → 录入数据库
  │   ├── 运营端维护中文
  │   └── 翻译服务填充英文（LLM辅助提词）
  │
  └── 模板文档 → 双版本维护
      ├── 中英文模板各自独立更新
      └── 变更时需同时修改两个版本
```

**代码规范**：
1. 所有用户可见字符串必须走 i18n，禁止硬编码
2. 日志/异常消息可中文，用户可见消息必须国际化
3. 数字/日期/货币格式按 Locale 自动格式化
4. 枚举值显示用 i18n message 映射

## 备选方案及舍弃原因

| 方案 | 舍弃原因 |
|------|---------|
| 全 DB 翻译表 + 运行时查询 | 每页需要 N 次 DB 查询，UI 静态文本用 DB 过度设计 |
| 全 .properties 文件 | 业务数据（产品名/合同条款）无法放在文件中 |
| 第三方翻译平台强依赖 | TMS API 不可用时系统不可用，需要离线回退 |
| 自动机器翻译全流程 | 法律/合同用机翻不可靠，需人工审核 |

## 影响

- **数据库设计**：多语言字段增加约 30% 列数，按 `_zh`/`_en` 命名
- **API 响应**：根据 Accept-Language 动态选择返回字段
- **前端开发**：所有文本需要 key 引用，禁止硬编码字符串
- **OpenClaw**：语言自适应逻辑，回复语言 = 用户输入语言 / 用户预设语言
- **QA 测试**：每个功能需覆盖中文和英文界面验证
- **部署**：.properties 文件需纳入 CI/CD 构建流程

## 后续演进

- Phase 2+: 接入翻译管理平台（TMS），建立翻译记忆库
- Phase 3+: 支持 Buye 自定义语言偏好，OpenClaw 对话语言自动匹配
- Phase 4+: LLM 辅助翻译审核，自动检查翻译完整性
