# ADR-013: Batch 11/12/13 批次规划决策

> **状态**: 🟢 Accepted | **日期**: 2026-06-12 | **关联**: ADR-001, ADR-003, ADR-010

---

## 背景

TradeOMS 采用批次交付模式（Batch），每个 Batch 覆盖一组功能模块和基础设施。Batch 11/12/13 是继 Batch 10（API 性能优化 + OpenAPI 集成 + 基础设施）之后的连续三个批次，需要明确划分边界。

核心决策问题：
- 基础设施（DB 迁移/索引/安全）与业务功能如何拆分
- 联调测试、E2E 测试、Figma 精修放在哪个批次
- 产品手册编写与开发的同步策略
- 软删除等横向能力如何跨批次落地

## 决策

### 批次范围总览

| 批次 | 代号 | 主线 | Agent 分工 | 落地情况 |
|------|------|------|-----------|---------|
| **Batch 11** | 基础设施 + KYC/CDP/Quote | DB 迁移 + 核心模块后端+前端 | 斗牛(后端) + 大鱼(前端+测试) + 巨贵(手册) + 金毛(E2E) | ✅ 已交付 |
| **Batch 12** | 优化 + VA/TT/LC | API 性能 + 软删除 + 索引完善 + OpenClaw 集成 | 斗牛(性能/安全) + 巨贵(手册+合同审查) | ✅ 已交付 |
| **Batch 13** | 精修 + 联调 + 手册 | Figma 精修 + E2E 全链路 + 产品手册补完 | 大鱼(Figma+E2E) + 斗牛(联调修复) + 巨贵(手册) | 🔄 进行中 |

### Batch 11 — 基础设施 + 核心模块

**交付范围**：

| 模块 | 后端 | 前端 | 手册 |
|------|------|------|------|
| 基础设施 | Flyway V1-V8 + Maven + Swagger | — | — |
| KYC | 创建/查询/提交 + 附件上传 | 表单 + 列表 | ✅ buyer + ops |
| CDP | 公司/部门/用户/角色 CRUD | 管理页面 | ✅ |
| Quote | 创建/查询/磋商/确认 | 报价列表+详情 | ✅ |
| Order | 创建/查询/合同 | 订单列表+详情 | ✅ 基本 |
| 安全 | JWT 认证 + Spring Security | 登录页 + 拦截器 | — |

**关键决策**：
- DB 迁移优先于功能开发（铁律：数据库第一优先级）
- 所有模块的 **Schema 设计 + 基础 CRUD** 在一个 Batch 完成
- JWT 认证在 Batch 11 落地，确保后续模块有认证基础

### Batch 12 — 性能优化 + VA/TT/LC + OpenClaw 集成

**交付范围**：

| 模块 | 内容 |
|------|------|
| 性能优化 | API 性能检查报告 + Flyway 索引 V9 + 慢 SQL 日志 |
| 软删除统一 | 全模块 is_deleted 字段统一 (V10) + 索引审计 (V11) |
| 审计日志 | V12 audit_log 创建 |
| 种子数据 | V13 参考数据 + V14 演示数据 + V15 字段修正 |
| VA (虚拟账户) | 后端 CRUD + 前端页面 + 运营管理 |
| TT (电汇) | 后端 CRUD + 前端页面 + 运营管理 |
| LC (信用证) | 后端 CRUD + 前端页面 + 运营管理 |
| OpenClaw 集成 | Buyer 端 AI 助手场景 + 消息推送 |
| 合同审查 | Buyer 端合同审查功能 + 前端实现 |
| 手册 | Buyer 端 + 运营端 VA/TT/LC 手册 |
| E2E 安全测试 | auth-security-e2e (登录/路由守卫/拦截器) |

**关键决策**：
- 性能优化先行（V9 索引），软删除紧随其后（V10）
- VA/TT/LC 三大支付模块在同一批次完成，保持一致性
- OpenClaw 集成不是替代前端，而是新增 AI 交互入口（Track A + Track B 共存）
- 产品手册随功能一起完成（不拖到后续批次）

### Batch 13 — Figma 精修 + E2E 全链路 + 产品手册补完

**交付范围**：

| 模块 | 内容 |
|------|------|
| Figma 精修 | UI 还原度审查 + 间距/颜色/响应式修复 |
| E2E 全链路 | 全链路 E2E 测试: KYC → CDP → Quote → Order → VA → TT → LC |
| 运营端手册 | 缺失的运营端手册补全 |
| 联调修复 | VT/TT/LC 联调发现的问题修复 |
| 涟漪扫描 | 代码涟漪扫描 + 规约扫描脚本自动化 |
| 质量门禁 | C.11.x 质量门禁完善 + 审计 |

**关键决策**：
- Figma 精修和 E2E 是模块验收必须步骤（铁律）
- 联调修复放在最后，确保所有模块可独立验证后再做集成验证
- 涟漪扫描脚本自动化：每次代码变更自动扫描受影响范围

### Agent 分工决策

| Agent | Batch 11 | Batch 12 | Batch 13 |
|-------|----------|----------|----------|
| 🐶 斗牛 | 后端基础设施 + 核心模块 CRUD | 性能+安全+索引 | 联调修复 + 门禁 |
| 🐋 大鱼 | 前端页面 + E2E 基础 | — | Figma 精修 + E2E 全链路 |
| 🐩 巨贵 | 产品手册 + 质量门禁 | 手册续写 + 合同审查 | 手册补完 + 涟漪扫描 |
| 🦮 金毛 | E2E 测试 | 安全审计 + E2E | — |
| 🧸 泰迪 | UI 精修 | — | — |
| 🐺 二哈 | CI 自动化 | — | — |

**规则**：
- 任务类型大于 Agent 名称（斗牛≠只能做后端，但按专长分配效率最高）
- 每个 Batch 至少两个 Agent 并行开发，一个负责 QA
- Agent 饱和派单：无空闲 Agent

## 影响

- **正向**：
  - 每 Batch 2-4 周，可快速验证和调整
  - 功能模块 vs 横向能力（索引/安全/手册）均衡分配
  - E2E 测试在每个 Batch 都落地，非一次性验收
- **约束**：
  - Batch 边界固定后不接受范围蔓延（新增需求 → 下一 Batch）
  - 依赖 Flyway 版本顺序，回退需要创建新迁移
  - 跨 Batch 的横向能力（如软删除）需要协调所有 Module Owner
