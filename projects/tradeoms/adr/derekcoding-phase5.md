## Phase 5 — 汲取营养

作为 derekcoding 的标准结束步骤，每次项目骨架搭完后，自动执行一轮 GitHub 开源对比。

### 搜索策略（覆盖全部业务模块）

| 轮次 | 搜索方向 | 关键词策略 | 目标项目数 |
|------|---------|-----------|-----------|
| 宽泛定位 | 领域整体项目 | `order management system`, `trade platform`, `commodity trading` | TOP 20 |
| 模块覆盖 | 逐个业务模块 | 模块名 + `system/management/platform` | 每模块 TOP 10 |
| 技术参考 | 架构/数据库/模式 | 具体技术关键词 | TOP 10 |

### 对比维度

| 级别 | 含义 | 处理方式 |
|------|------|---------|
| 🔴 架构级 | 影响整体架构或核心设计方向 | 必须问你意见 |
| 🟡 功能级 | 可补充到模块文档的优化点 | 直接更新，记录来源 |
| 🟢 验证级 | 确认我们决策合理的证据 | 记录到 ADR 加固 |

### 输出

- 对比报告 → `09-migrations/github-comparison-round{n}.md`
- 架构级差异 → 单独问你
- 功能级差异 → 直接更新对应文档
- 验证级发现 → 备注到对应 ADR

## 配套方法论：内部涟漪分析

Phase 5 是「向外看」，从外部汲取灵感后引入新规约。

引入后必须执行 **内部涟漪分析**（[spec-ripple-analysis.md](../12-development-guide/spec-ripple-analysis.md)），
系统性检查 OMS 所有相关文件（ADR、runbook、开发流程、Phase 规划等）是否需要配套更新。

```
外部获取 → Phase 5: 开源对比 / 行业最佳实践
    ↓  发现值得引入的规约
内部落地 → 涟漪分析: 逐层扫描三层文档模型
    ↓  逐文件更新
一致性 → 更新 ADR + runbook + dev流程 + Phase规划
```
